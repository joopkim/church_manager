# Tech Stack

## 확정 스택

창원 초대교회 소그룹 관리 웹앱은 다음 기술 스택으로 개발한다.

```text
App Framework: Next.js
Language: TypeScript
Styling: Tailwind CSS
Database: Supabase PostgreSQL
Authentication: Supabase Auth
Authorization: Supabase Row Level Security + App-level checks
Hosting: Vercel
Storage: Supabase Storage, 필요 시
```

## 선택 이유

이 서비스는 클라우드에 배포되어 관리자, 리더, 성도가 접속하는 웹앱이다. 성도 개인정보, 구역/사역팀 소속, 소그룹 모임 기록, 출석, 기도제목 등 관계형 데이터와 권한 관리가 중요하다.

따라서 다음 조건을 만족해야 한다.

- 클라우드 배포가 쉬워야 한다.
- PostgreSQL 기반 관계형 데이터베이스를 사용해야 한다.
- 인증과 권한 구조를 빠르게 구성할 수 있어야 한다.
- 초기 운영 비용과 유지보수 부담이 낮아야 한다.
- 향후 예배, 기도회, 봉사 배정, 통계 기능으로 확장 가능해야 한다.

위 조건을 기준으로 `Next.js + Supabase PostgreSQL + Supabase Auth + Vercel` 조합을 1차 개발 스택으로 확정한다.

## 구성 요소별 역할

### Next.js

역할:

- 웹앱 프레임워크
- 관리자/리더/성도 화면 구성
- 서버 컴포넌트, 서버 액션, 라우트 핸들러를 통한 서버 측 검증
- Vercel 배포와의 높은 호환성

선택 이유:

- React 기반으로 화면 개발 생산성이 높다.
- 프론트엔드와 서버 로직을 한 프로젝트 안에서 관리할 수 있다.
- 관리자 화면, 대시보드, 상세 화면 같은 웹앱 구조에 적합하다.

### TypeScript

역할:

- 데이터 모델과 UI props의 타입 안정성 확보
- 권한/상태/출석 값의 실수 방지
- 장기 유지보수성 향상

선택 이유:

- 성도, 구역, 사역팀, 모임, 출석처럼 관계가 많은 도메인에 적합하다.
- 리팩터링과 확장 시 안정성이 높다.

### Tailwind CSS

역할:

- 디자인 가이드의 색상, 간격, 타이포그래피 토큰 반영
- 빠른 UI 구현
- 반응형 화면 구현

선택 이유:

- 디자인 시스템을 코드에 빠르게 적용할 수 있다.
- 운영 도구형 화면의 표, 필터, 패널, 배지 구성에 적합하다.

### Supabase PostgreSQL

역할:

- 핵심 데이터 저장
- 성도, 구역, 사역팀, 리더 범위, 소그룹 모임, 출석 기록 관리
- SQL 기반 검색, 필터, 통계 확장

선택 이유:

- PostgreSQL 기반이라 관계형 데이터 구조에 적합하다.
- 클라우드 DB 운영 부담이 낮다.
- Supabase 콘솔에서 데이터와 SQL을 관리할 수 있다.

### Supabase Auth

역할:

- 사용자 로그인
- 관리자/리더/성도 계정 관리
- 세션 관리

선택 이유:

- 인증을 직접 구현하는 부담을 줄인다.
- Supabase PostgreSQL 및 RLS와 자연스럽게 연결된다.

### Supabase Row Level Security

역할:

- DB 레벨 접근 제한
- 성도는 자기 정보 중심 접근
- 리더는 담당 구역/사역팀 범위 접근
- 관리자는 전체 접근

선택 이유:

- 개인정보 보호에 유리하다.
- 앱 코드 실수로 인한 과도한 데이터 노출을 줄일 수 있다.

주의:

- RLS 정책은 반드시 테스트해야 한다.
- RLS만 믿지 않고 Next.js 서버 코드에서도 역할과 범위를 재검증한다.

### Vercel

역할:

- Next.js 앱 배포
- HTTPS 제공
- 환경변수 관리
- 배포 프리뷰와 운영 배포 관리

선택 이유:

- Next.js와의 호환성이 높다.
- 초기 배포와 운영이 쉽다.
- 작은 팀이나 개인 개발자가 관리하기 좋다.

## 권한 설계 원칙

권한은 두 층으로 확인한다.

```text
1. DB 레벨: Supabase RLS
2. 앱 레벨: Next.js 서버 코드
```

역할:

- `member`: 자기 정보와 자기 소속 중심 접근
- `leader`: 담당 구역/사역팀 범위 접근
- `admin`: 전체 관리

리더 권한은 단순 role만으로 판단하지 않는다. 반드시 `leader_scopes`를 확인한다.

예시:

- 3구역 리더는 3구역 성도와 3구역 모임을 관리할 수 있다.
- 찬양팀 리더는 찬양팀 구성원을 관리할 수 있다.
- 한 사람이 구역 리더이면서 사역팀 리더일 수 있다.

## TCO 판단

TCO는 총소유비용 관점에서 판단한다. 여기에는 월 비용뿐 아니라 개발 시간, 유지보수, 보안, 백업, 장애 대응, 확장 비용이 포함된다.

### 확정 스택의 장점

- 초기 개발 비용이 낮다.
- 인증과 DB 운영 부담이 낮다.
- PostgreSQL을 직접 사용할 수 있다.
- Vercel과 Supabase의 무료/저비용 구간에서 시작할 수 있다.
- 운영자가 서버를 직접 관리하지 않아도 된다.
- 향후 데이터가 늘어도 PostgreSQL 기반으로 확장 가능하다.

### 확정 스택의 비용 요소

- Supabase 사용량 증가 시 비용 확인이 필요하다.
- Vercel 사용량 증가 시 비용 확인이 필요하다.
- RLS 정책 설계와 테스트에 시간이 필요하다.
- 특정 플랫폼에 대한 의존성이 생긴다.

### 후보 B 대비 판단

후보 B:

```text
Next.js + Prisma + Managed PostgreSQL + Auth.js
```

후보 B는 데이터 모델을 Prisma schema로 명확히 관리할 수 있고, 앱 서버 중심으로 비즈니스 로직을 통제하기 좋다. 또한 특정 BaaS에 대한 의존도가 낮다.

하지만 첫 개발과 운영에서는 다음 부담이 있다.

- 인증, 권한, DB, 백업, 배포를 더 직접 조합해야 한다.
- 권한 검증을 앱 코드에서 더 많이 책임져야 한다.
- 초기 개발 속도가 느릴 수 있다.
- 운영자가 이해해야 할 구성 요소가 많아진다.

현재 프로젝트는 교회 내부 운영을 위한 1차 MVP가 중요하므로, 후보 A가 TCO 관점에서 더 적합하다.

## 보안 기준

필수:

- 모든 운영 접속은 HTTPS를 사용한다.
- 환경변수는 코드에 커밋하지 않는다.
- 성도 개인정보는 권한별로 제한한다.
- 리더는 담당 범위 밖의 성도 데이터를 볼 수 없어야 한다.
- 관리자 기능은 서버 코드에서 role을 재검증한다.
- RLS 정책은 테스트 데이터로 검증한다.
- 삭제는 가능한 한 물리 삭제보다 비활성 처리한다.

권장:

- 변경 이력을 남긴다.
- 민감한 메모는 별도 권한 정책을 둔다.
- 정기 백업 정책을 마련한다.
- 운영 전 개인정보 처리 기준을 문서화한다.
